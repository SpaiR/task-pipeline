# Invariants — don't break these when editing skills

## Universal

- **Hard-stop preconditions are duplicated at the bash layer on purpose** (context scripts + `validate.sh` + `close.sh` each enforce independently). The **prompt-layer** preamble is shared via the [Shared prompt preamble](#shared-prompt-preamble) section below (three profiles A/B/C by code-navigation tier) — modifying those profiles does **not** relax the bash gates. Don't "DRY" the bash duplication away — fail closed at each step independently.
- **`validate.sh` is a precondition gate, not optional.** Context scripts and `close.sh` invoke it after the `config.md` check; orchestrator skills (`/task:design`, `/task:build`) call it inline at Step 0. It checks: `task.md` line-1 header, `---` separator, `## Description`; soft-warns when `Roadmap:` is present without matching `Source item: #<N>` (auto-mark would silently skip that umbrella in `/task:ship`); per-step non-empty `Goal:` and `Touches:` (no `...`) in `plan.md`; English `### Context`/`### Goal`/`### Outcomes`/`### Acceptance criteria` blockquote sub-headings in `.task/roadmap/*.md`. Don't bypass — downstream parsers (`close.sh`, `design`'s open phase `--from` parser) silently misbehave on malformed artifacts. The `PreToolUse` hook (`hooks/hooks.json`) enforces it at runtime as defense-in-depth.
- **Append-only artifacts.** `## Iteration N` in `audit.md` never touches N-1. `## Decisions` in `task.md` (architect/Socratic-mode idea phase) and `plan.md` (refine phase) are append-only too — record only decisions that *changed* or *clarified non-obvious* choices, not confirmations of defaults.
- **Pipeline is invisible to the project.** All tracked state lives under `.task/`. Two sanctioned artifacts may sit outside `.task/`: (1) `.task-current` (one-line per-worktree pointer at the worktree root, written by `/task:design`'s open phase initial mode and — under `/task:auto-roadmap` — by the first item-runner's design-runner; removed by a default `/task:ship` full close, also reachable via the `--full` alias); (2) a `.task` **symlink** to the main worktree's `.task/`, present only in a linked git worktree and materialized solely by `/task:bootstrap` Step 0 join-mode — a passive artifact (no executable code), the sanctioned way separate worktrees share one `.task/`. `/task:bootstrap` must not modify `CLAUDE.md`, `README.md`, `.gitignore`, or any other shared file; both `.task` and `.task-current` are excluded locally through `.git/info/exclude` — the exclusion pattern is `.task` **without a trailing slash** (a slash matches only a directory and would miss the worktree symlink). `.task-current` is never symlinked.
- **`WS_DIR` is resolved through `skills/_lib/resolve-ws.sh`, not constructed inline.** Every bash helper that needs `WS_DIR` (`*-context.sh`, `validate.sh`, `close.sh`) sources the resolver — in practice via `skills/_lib/preamble.sh` (`source_resolve_ws`). `validate.sh` is the one exception: it sources `_lib/resolve-ws.sh` directly because its `all` subcommand uses a best-effort, stderr-quiet `resolve_ws 2>/dev/null` form. The resolver reads `$TASK_ID_OVERRIDE` (env) > positional arg > `.task-current`, and fails loud on missing / stale state. The legitimate exception is `auto-roadmap-context.sh`, which operates on the umbrella container before any `.task-current` exists; it calls `preamble.sh`'s `set_workspace_root` (semantically distinct — `WS_DIR=.task/workspace`).
- **`${CLAUDE_SKILL_DIR}` is a Claude Code load-time substitution, not a shell env var.** It is expanded inside `SKILL.md` / `phases/<phase>.md` content before the model sees the prompt (alongside `${CLAUDE_SESSION_ID}` / `${CLAUDE_EFFORT}` / `$ARGUMENTS`); the resulting absolute path lands in the prompt and the model just runs the bash line as-is. **Do not** instruct the model to prepend an inline `CLAUDE_SKILL_DIR=…` to a same-line `bash "${CLAUDE_SKILL_DIR}/…"` invocation — bash expands the variable in the parent shell *before* the inline assignment takes effect, leaving an empty path (e.g. `bash "/audit-context.sh"` → exit 127). Each skill file with such an invocation carries a `Run verbatim.` callout next to the bash block; preserve that callout. If a future Claude Code release exposes `${CLAUDE_SKILL_DIR}` as a runtime shell env var instead of a load-time substitution, this rule can soften — until then, the prescribed wrapping for an unsubstituted fallback is `bash -c '…'` (assignment + inner expansion happen in the subshell, in order). One non-obvious unsubstituted case: `/task:auto-roadmap`'s `auto-roadmap-item-runner` runs the build audit phase **inline** (it `Read`s `phases/audit.md` directly — no skill load → no substitution). The fallback must resolve the script at the **build skill root** (`${CLAUDE_PLUGIN_ROOT}/skills/build`), NOT the phase file's own directory (`skills/build/phases`) — `audit-context.sh` sits at the skill root, so guessing the path from where `audit.md` lives yields `phases/audit-context.sh` → exit 127. (`${CLAUDE_PLUGIN_ROOT}` is available in subagents, so the item-runner resolves it the same way the driver's main thread used to.)
- **Roadmap-touching helpers go through `skills/_lib/roadmap.sh`.** `resolve_roadmap_path`, `roadmap_mtime`, `roadmap_progress_counts`, `list_roadmap_items` are defined once there; do not re-implement them inline. Today's callers: `auto-roadmap-context.sh`, `validate.sh`.
- **`/task:ship` stages only task-related files; never anything under `.task/`** (working artifacts under `workspace/<task-id>/` are archived by ship's close step into `log/`; neither subtree gets staged). `.task-current` is also never staged (excluded via `.git/info/exclude`).
- **`disable-model-invocation: true` + `user-invocable: true`** on every skill — the model never auto-invokes; users type `/<slug>`. Exception: `validate` runs `user-invocable: false` (internal utility — invoked from context scripts and the PreToolUse hook).
- **Language rules live in `config.md` → "Language", not in skills.** All artifacts follow the language of `task.md` Description: `task.md`, `plan.md`, `audit.md`, `summary.md`, all `## Decisions`. User-facing dialog in design/idea, design/refine, `/task:roadmap`, `/task:auto-roadmap` (wizard, confirm, final report) follows the same; design/idea phase architect mode and design/open phase quick-draft both fall back to the language of `$ARGUMENTS` when Description is being bootstrapped from empty. **Subagent prompts must propagate language explicitly** — build's audit phase per-call template carries an explicit `--- Language ---` block; agents translate the **values** of `category`/`problem`/`fix` while keeping field keys, the `high`/`med`/`low` enum, file paths, and the `no findings` sentinel English (parser-stable). Same English-skeleton/translate-on-render rule applies to `audit.md` template headers. **Runner return strings (`OK: ...` / `FAIL at <stage>: ...`) and `workspace/<task-id>/auto.lock` / `auto-error.log` contents stay English regardless** — they are parser-stable and consumed by the orchestrator.

## Three code-navigation tiers

Moving a skill between tiers must be deliberate. The allowed/forbidden reads per tier are formalized in the [Shared prompt preamble](#shared-prompt-preamble) section below (Tier A / B / C). Each `SKILL.md` references its profile.

- **No code navigation:** `/task:ship`, `validate`, `/task:auto-roadmap`, `/task:bootstrap`. Allowed reads: `CLAUDE.md`, `.task/config/config.md`, `.task/workspace/<task-id>/*.md`, `.task-current`, `.task/log/**/*.md`, `.task/roadmap/*.md` (read-only for design's open phase `--from`; read + targeted in-place flip for ship's close step auto-mark), `git`. `/task:ship` additionally consumes any project commit-format doc (e.g. `CONTRIBUTING.md`) when `config.md` → "Commit Format" points at it via `**Source:** <path>`. `/task:auto-roadmap` is here because its SKILL.md (the driver) only orchestrates — all code work delegates to the `auto-roadmap-item-runner` subagent (which in turn spawns the design/build runners + lens auditors and runs `/task:build` audit / `/task:ship` inside its own context). **Design's open phase is mixed-tier** — the header-only path (`--idea` opt-out / empty fresh call) is Tier A; the quick-draft path is Tier C (see next bullet).
- **Shallow scan + project docs:** `/task:bootstrap`, `/task:roadmap`, design's idea phase, design's open phase **quick-draft path only**. Manifests, top-level dirs (1–2 levels), `docs/`. No source files, no `Grep` over code, no MCP code-navigation. (`/task:bootstrap` also reads `CLAUDE.md` and `git log` to seed `config.md`. Open's quick-draft uses the same shallow-scan allowance as idea phase's architect mode Step A.1 — top-level dirs, build/manifest files, `CLAUDE.md` — to ground the paraphrase in the project's stack.)
- **MCP-first:** design's blueprint + refine phases, `/task:build` (both phases: implement, audit). Use MCP tools from `config.md` in priority order; built-in `Grep`/`Glob`/`Read` are fallback-only.

`/task:design` and `/task:build` are mixed-tier orchestrators — each phase has its own tier (declared inside the companion phase file). Design's open phase itself is internally mixed (A on the `--idea` header-only path, C on quick-draft). The orchestrator SKILL.md itself does config gate + phase detection + dispatch + (build) auto-fix loop + (design) a bounded idea-phase elicitation + open→idea chain on a fresh brainstorm start, all of which falls under Tier A.

## Agent class — auditor + narrow executors

`agents/` ships **six** auditor-class agents (three for `/task:build` audit phase, three for `/task:roadmap --refine`) and **three** executor-class agents for `/task:auto-roadmap`'s per-item cycle (`auto-roadmap-item-runner` and the two narrow-scope runners it spawns).

- **Auditor-class agents** (build-audit lenses: `agents/audit-{reuse,simplicity,clarity}-auditor.md`; roadmap-refine lenses: `agents/audit-roadmap-{coverage,decomposition,clarity}-auditor.md`) MUST declare a read-only `tools:` allowlist (currently `Read, Grep, Glob`) — never `Edit`/`Write`/MCP-edit. Read-only is **runtime-enforced** by Claude Code from the frontmatter, so the allowlist is the actual safety guarantee. Adding `Edit` or `Write` to an auditor's `tools:` is a hard violation. Shared prompt-layer rules live in [`agents/_shared/audit-rules.md`](../../agents/_shared/audit-rules.md) and apply to **both** families (build-audit operates on the code diff; roadmap-refine on the roadmap file — substitute "diff location" with "roadmap location" when reading the shared rules). The `tools:` allowlist stays in each agent's frontmatter.
- **Executor-class agents** (`agents/auto-roadmap-item-runner.md`, `agents/auto-roadmap-design-runner.md`, `agents/auto-roadmap-build-runner.md`) intentionally declare no `tools:` — all three inherit the full toolset from the parent interactive session. None declares `model:` in frontmatter: the item-runner and design-runner inherit the parent-session model (typically opus for design work); build-runner's model is set per-spawn by the item-runner via `Agent.model` override, sourced from each item's `plan.md → Implement-Model:` (`opus|sonnet|haiku`). The split exists so implement can run under a cheaper model than design — see [auto-roadmap.md § Per-stage model split](auto-roadmap.md). Scope:
  - **item-runner** — the whole per-item cycle: spawns design-runner + build-runner, fans out the three lens auditors, runs audit + ship inline, returns a compact report-card digest;
  - **design-runner** — design's open + blueprint phases only (spawned by item-runner);
  - **build-runner** — build's implement phase only (spawned by item-runner).

  **design-runner and build-runner stay leaves** — do not let either spawn subagents or swallow audit/ship; keeping them scoped is what lets their heavy per-phase context (blueprint code-nav, implement diff) be discarded on return rather than lingering in the item-runner. Audit's lens fanout and ship live one level up, in the item-runner (nested subagent spawning is supported — this is a scoping choice, not a platform limit). Shared registry of rules inherited from nested phase files lives in [`agents/_shared/runner-rules.md`](../../agents/_shared/runner-rules.md) — edits to a source-of-truth phase file (e.g. `build/phases/implement.md`) must update the shared file in the same commit.
- **Adding a new agent must explicitly state its class** in the description and follow the matching `tools:` rule. Mixing roles (e.g. an "auditor that also writes a fix file") needs a deliberate redesign — usually splitting it into a read-only audit agent + a main-thread fix step.

## Per-skill

### `/task:design`

- **Thin orchestrator over four phase companion files.** `skills/design/SKILL.md` does config gate (Step 0) + phase detection via `_lib/phase-detect.sh` (Step 1) + dispatch (Step 2 — read `skills/design/phases/<phase>.md` verbatim; `PHASE=idea` on a fresh start chains open(header-only) → idea(architect) in one call) + chain hint (Step 3). The phase files (`open.md`, `idea.md`, `blueprint.md`, `refine.md`) carry the actual per-phase Steps.
- **Phase detection is auto-detect by default, `--phase` override.** Without flags: `phase-detect.sh design` reads workspace state and returns one of `open|idea|blueprint|refine-prompt`. The `refine-prompt` branch does NOT auto-enter refine — it surfaces a chain hint and stops; refine requires explicit `--phase refine` (or `--refine` shorthand). An empty `/task:design` call returning `open` with no positional context is re-routed to `idea` (brainstorm) by the orchestrator.
- **Idea phase mode is a function of `## Description` content, with one roadmap-mode short-circuit.** Empty → architect mode: 2–3 concrete directions per round, explicit recommendation, specific risks, one focused question; appends a top-level `## Decisions` section. Non-empty → Socratic mode: questions about completeness/clarity/assumptions/blind-spots/idea-critique; rewrites Description; appends top-level `## Decisions` (warns first if `plan.md` already exists). Roadmap-mode exception: if the header carries `^Roadmap: ` AND Description is empty, idea phase stops with a redirect to `/task:design --from <roadmap-slug>`.
- **Blueprint phase is the single source of `## Steps` three-layer contract.** Each step has `Goal` (detailed intent and expected end state — not shortened to one line), `Touches` (concrete symbols, full names, no `...` placeholders), and optional `Logic` (pseudocode, only when branching/flow is non-obvious). Load-bearing: implement phase keys off `Goal` and restricts edits to `Touches`; audit phase reads `Touches` for scope-gate enforcement; refine phase must preserve all three when rewriting. If `Logic` conflicts with `Goal`, `Goal` wins.
- **Blueprint reads pinned spec decisions (Step 1.5).** When `task.md` carries a `Roadmap:` header and `## Description` cites `<slug>.spec.md §N` under `### Spec references`, blueprint resolves the sidecar with the **same rule as the Clarity auditor** (directory from the `Roadmap:` line, filename taken verbatim from the citation token — not reconstructed from the roadmap basename, so both readers hit the same file even after a hand-edit), reads the cited `## N.` sections, and treats them as fixed anchors — `## Steps` honors them rather than re-deriving a different choice; everything the sidecar does not pin stays blueprint's free choice. Missing file/section → interactive **stop and ask**, non-interactive (`--auto` / `/task:auto-roadmap`) **WARN and proceed** on the behavioral Description alone (reference integrity is a Clarity-auditor concern, not a blueprint hard-stop; a dangling ref is never a runner FAIL). This is the consumer half of the spec sidecar — without it the `### Spec references` pointer is decorative.
- **Testing Policy → `## Tests` is the single source of truth.** `config.md` → `Testing Policy` → `Mode` is `always` / `on-demand` / `never`. Blueprint phase resolves `tests_required` once (on-demand: only if `task.md` Description explicitly asks). If true, blueprint emits `## Tests`; the implement phase keys off the **presence of that section**, never re-reading `Testing Policy`.
- **Open phase refuses if `.task-current` exists**, except for the from-roadmap continuation path. The single relaxation: `--from <pathOrSlug>[#<N>]` is allowed when (a) Description body is empty, (b) header has `^Roadmap: <P>$`, (c) `<P>` resolves to the same roadmap as the new arg, (d) line-1 task-id matches. In that case open rewrites only the per-subtask `Source item:` line and Description body, preserving line 1, `Roadmap:`, `.task-current`, and any `## Decisions`.
- **Open phase (Mode 1, manual) writes Description in the same call via quick-draft.** Step 2a decides between two sub-paths on a fresh open (no `.task-current`):
  - **Quick-draft (default).** Any non-empty **paraphrasable** context. Tier C shallow scan (top-level dirs, manifests, `CLAUDE.md`) + single-pass paraphrase of the provided context into `### Problem` / `### Outcome` / `### Scope` / `### Constraints` sub-sections inside `## Description`. Next `/task:design` auto-enters blueprint. There is no tiny-input token-count heuristic — `[DT-5177] fix login` is paraphrased into a minimal Description; only input with no prose at all (a bare ticket id) falls to the header-only path below.
  - **Header-only.** Description stays empty (Tier A). Two triggers: (1) `--idea` opt-out — the flag is mutually exclusive with `--from`, `--phase`, `--refine` (orchestrator validates in Step 1); the orchestrator then continues into the idea phase (architect mode) **in the same call** — open(header-only) → idea(architect). An empty `/task:design` with no task in flight routes here (orchestrator elicits the idea first). (2) **No paraphrasable prose** — input is only a ticket id; there is nothing to quick-draft, so Description is left empty and the **next** `/task:design` auto-enters the idea phase (architect). Header + `.task-current` creation stays owned by open in both.
- **Open phase (Mode 2, from-roadmap) derives task-id by invoking `_lib/derive-task-id.sh`** — single source of truth for the algorithm.
- **`Implement-Model:` stamp in `plan.md`.** Blueprint phase Step 3 stamps `Implement-Model: <opus|sonnet|haiku>` between `# Plan:` and `## Scope` per a complexity rubric. The stamp is parser-validated by `validate.sh` and **load-bearing for `/task:auto-roadmap`** — the `auto-roadmap-item-runner` reads it (its Step 3) after its design-runner OK and passes the value as `Agent.model` override when spawning `auto-roadmap-build-runner`. Harmless in manual flows where no runtime consumer keys off it. Position is convention-only; `validate.sh` matches the line anywhere in the file.

### `/task:build`

- **Thin orchestrator + bounded auto-fix loop.** `skills/build/SKILL.md` does config + artifact gates (Step 0) + argument parsing (Step 1) + phase detection via `_lib/phase-detect.sh` (Step 1a) + optional `--auto` budget gate (Step 1b) + dispatch (Step 2 — read `skills/build/phases/<phase>.md` verbatim for implement; for audit, wrap in the loop at Step 4) + chain hint or `--auto` loop-back (Step 5). The phase files (`implement.md`, `audit.md`) carry the per-phase Steps; audit's auto-fix loop semantics live in SKILL.md because that's control flow, not phase logic.
- **`--auto` is opt-in one-shot mode.** Default `/task:build` runs **one** phase per invocation and prints a chain hint; this gives a human checkpoint between phases (`implement` → `audit` reads the diff directly). Passing `--auto` replaces the chain hint at Step 5 with a loop-back to Step 1a, chaining `implement → audit` in a single invocation. Mutually exclusive with `--phase` (Step 1 rejects loud). Per-phase budget gate in Step 1b — symmetric with audit's own bounded loop:
  - `implement` — in-memory counter `IMPLEMENT_DISPATCHED`, max 1 dispatch per `--auto` invocation. Re-dispatch implies the user reverted the diff mid-loop, which `--auto` treats as out-of-band intervention → stop.
  - `audit` — max `N` across `^## Iteration N` headers in `"$WS_DIR/audit.md"` (0 if missing; symmetric with `audit-context.sh`'s `max(## Iteration N) + 1`), max 2 iterations total. Belt-and-suspenders with Step 4's own internal `passes_done < 2` bound: Step 4 caps **per-invocation** passes, Step 1b caps **cross-invocation** re-entries (audit could otherwise be re-dispatched after `passes_done` resets between `--auto` loop iterations).
  Budget exhaustion → stop with `--auto stopped: <reason>. See $WS_DIR/<artifact>. Run /task:build --auto again to retry after manual investigation, or /task:design --refine to revisit scope.` Reasons + artifact paths are **English** (parser-stable) so future tooling can pattern-match them. Counts are read from on-disk artifacts so the budget survives session restarts.
- **Implement phase materializes plan steps as `TaskCreate` and verifies before completion.** Each `### Step N` of `plan.md` becomes one `TaskCreate`; the step closes with mandatory `Identify → Run → Read → State` verification (`Touches` symbols visible in `git diff`, RED→GREEN if `tests_required`); only on pass does `TaskUpdate(completed)` fire. Re-runs read `TaskList` and resume from the first non-`completed` step.
- **Implement phase: at most one quick-fix on failure, then hands off.** If build/tests fail and the cause is obvious from output (typo, missing import, wrong symbol name in `Touches`), apply a single targeted fix and re-run. Otherwise — stop and report. No shotgun fixes. Systematic root-cause investigation is **the user's responsibility**.
- **Audit auto-fix is bounded ≤2 iterations, runs in main thread, scope-gated.** Build's SKILL.md Step 4 wraps audit's lens fanout in a loop: spawn 3 lens-agents (via `Agent(..., context: fork)`) → merge findings → apply fixes in main thread severity order. Each fix is gated through `bash skills/_lib/touches-gate.sh plan.md` — out-of-scope fixes (files outside the whitelist derived from `plan.md → File:` and `→ Touches:`) are marked `Skipped: out-of-scope (touches gate)` and not applied. The gate strips backticks / parenthesized descriptions / trailing prose after em-/en-dash (and after a `: ` colon-plus-space, which leaves a `path:line` suffix intact) before resolving each token, normalizes both plan paths and `git diff` output to repo-relative form so an absolute `File:` path still matches the relative diff entry, and emits a stderr `WARN:` for any token that resolves to zero files so malformed entries surface instead of silently shrinking the whitelist. After 2 iterations with pending high-severity findings, surface to user and stop.
- **Audit merge runs three filter gates before write — hunk → claude_md_quote → confidence.** Inside Step 3 of the audit phase file, after dedup + existing-symbol filter, findings pass through three additional gates: (3a) **hunk-gate** drops any finding whose `location` line is not inside an added or modified hunk of the file (parsed from `@@ -<old> +<new_start>,<new_count> @@` headers in the `diff bundle`); (3b) **claude_md_quote-gate** (Clarity-only) drops findings citing a CLAUDE.md convention when the mandatory `claude_md_quote:` field is missing or its phrase is not verbatim in the `--- CLAUDE.md ---` block passed to the agent (case-insensitive, whitespace-collapsed substring match); (3c) **confidence-gate** drops `med`/`low` findings scored `<75` by the merger using a 0/25/50/75/100 rubric (high-severity bypasses this gate). Dropped findings land in an optional `### Filtered (low confidence)` block inside the iteration — one bullet per drop with the cause `(line not in diff hunks)` / `(claude_md_quote not found)` / `(confidence: <N>)`. The Filtered block carries **no `Status:` lines** so `phase-detect.sh`'s whole-file `pending fix` grep treats it as inert; the iteration's `### Result` line tallies `— filtered: K`. If every finding is filtered, `### Findings` keeps only the header, `### Details` is omitted, and `phase-detect.sh` correctly classifies the phase `done`.
- **Audit phase orchestration is adaptive.** `audit-context.sh` emits `trivial: true|false` (1 file AND <30 changed lines → trivial). The phase file's Step 2 takes the inline combined-lens path when `trivial: true`; otherwise it fans out to three named lens agents in parallel. The agents are bundled with the `task` plugin (files under `agents/`) with read-only `tools:` allowlists — read-only is **runtime-enforced**, not just textual. Plugin prefix on `subagent_type` is mandatory (`task:audit-reuse-auditor` / `-simplicity-auditor` / `-clarity-auditor`); unprefixed names do not resolve and the runtime silently falls back to the catch-all `claude` agent (= zero tool uses, lens prompts dropped). The context script additionally emits a `recent history` section (`git log -5 --oneline` per changed file) consumed by Simplicity ONLY — surfaces churn signals like "defensive check just removed, now being re-added" that read as dead code only against the historical axis; like CLAUDE.md it must be dropped from the prompts built for Reuse and Clarity.
- **Don't silently fall back to inline prompts if agents are missing** — surface the install error; falling back loses the runtime-enforced read-only allowlist.
- **Audit does not touch `summary.md`.** It is owned by the implement phase. If an audit fix would change user-visible behavior, surface it as out-of-scope (the touches-gate also catches this when the change escapes the planned `Touches`).

### `/task:ship`

- **Two modes — full close (default) and subtask transition (`--next`).** The default (full close) archives per-subtask artifacts (`plan/audit/summary.md`) **plus `task.md`**, then removes the entire `workspace/<task-id>/` subfolder (sweeping orchestrator state like `auto.lock` / `auto-error.log` together with it) **and** removes `.task-current`. `--full` is accepted as a backward-compatible alias of this default. `--next` archives the per-subtask artifacts only, keeps `task.md` in `workspace/<task-id>/` with the **body of Description** cleared, and leaves `.task-current` untouched; the header and any `## Decisions` survive the transition. `--next` requires non-empty Description; the default full close allows empty Description.
- **Slug is optional and generated by the skill, not by `close.sh`.** When `$ARGUMENTS` carries no slug, ship's SKILL.md derives one from `summary.md` (primary) or `task.md` Description (fallback) before invoking the script. Slug is always English kebab-case `{feat,fix,chore}-{3-4-words}` regardless of `config.md` → "Language" — it's a filesystem identifier.
- **Auto-marks the source roadmap.** When Description is non-empty AND header carries both `Roadmap: <path>` and `Source item: #<N> — <title>` lines, `close.sh:Step 1.5` flips `### - [ ] <N>. ` → `### - [x] <N>. ` in `<path>` before archiving. **Loud failure** (exit 1, archive not run) on stale paths, missing item, or item without a checkbox. This is the **only** roadmap mutation the pipeline performs.
- **Commit format comes from `config.md` → "Commit Format".** When that section emits `**Source:** <path>` (typically `CONTRIBUTING.md`), `commit-context.sh` bundles the referenced doc into its output. `skills/_lib/templates/conventional-commits.md` is fallback only when neither `config.md` nor a referenced doc specifies a format.

### `/task:roadmap`

- **Blockquote sub-headings stay English regardless of `config.md` language** — `### Context`, `### Goal`, `### Outcomes`, `### Invariants`, optional `### Contracts`, `### Acceptance criteria`, optional `### Spec references`. Renaming or translating breaks both the design open phase `--from` parser and `validate.sh`. `Outcomes` / `Goal` / `Invariants` / `Contracts` are **behavioral**: they name observable properties of the system or world, not project-specific files / types / functions / constants. Normative names from the project's spec or `CLAUDE.md` are permitted — they address shared concepts. The hard cutover from the legacy `### Changes` heading happened in v0.4 (no aliasing); active downstream roadmaps still on `### Changes` migrate with `sed -i 's/^> ### Changes$/> ### Outcomes/' <roadmap>`.
- **`### Context` precedes `### Goal` inside the blockquote** and is mandatory — it is the only per-task motivation/"why" field that propagates into `task.md` via `/task:design --from`. Distinct from Goal: Context answers "why this task, what it unblocks"; Goal is the target state. Validator enforces presence.
- **Sizing is by `### Outcomes` bullet count**, not by file count or hours: `small` = 1–2, `medium` = 3–6, `large` = 7+. Decomposition auditor flags items where `Size:` and the actual outcomes count disagree, and items with ≥ 7 outcomes or outcomes spanning ≥ 2 unrelated behavioral domains as compound. Implementation overlap (same files / modules between items) is **not** a roadmap-level signal — that lives below this layer.
- **`**Class:**` per-task field is a best-effort hint**, not a validated contract. Closed-list shape (`rote-refactor | new-substrate | cross-module-migration | product-feature | content-vocabulary | tooling`) helps downstream rubrics — `/task:design`'s `Implement-Model:` selection, clarity auditor's `missing contracts` check for substrate classes — read intent without re-deriving it from prose. Empty / off-list values are tolerated by `validate.sh`.
- **Optional spec sidecar `<slug>.spec.md` is brainstorm-owned and behavioral-discipline's pressure-release valve.** Written only in brainstorm mode (Step 5–7), only when the discussion produced load-bearing technical decisions; no anchors → no file. It holds numbered `## N.` decision sections (`**Decision:**` / `**Rationale:**` / `**Constrains:**`) so the roadmap proper can stay behavioral — the *why* lives in the sidecar instead of leaking project-specific names into `### Outcomes` / `### Goal`. Items cite sections via `### Spec references` → `<slug>.spec.md §N`. **Boundary test (load-bearing — get it wrong and the spec degrades into a duplicate `plan.md`):** a decision belongs in the sidecar iff (a) blueprint re-deriving freely could pick differently AND (b) that divergence would break cross-item consistency or distort the initiative; file layouts / signatures / per-step lists stay in per-task `plan.md`. `### Contracts` (behavioral boundary shape) and the sidecar (technical decision behind it) are complementary, not duplicates. Structural labels English, prose config-language. `validate.sh` does not enforce it; the Clarity auditor flags dangling `§N` references (`broken spec ref`); refine never writes it.
- **`--refine` is phase-file dispatch, never auto-entered.** `roadmap/SKILL.md` Step 0a routes on the literal `--refine` token in `$ARGUMENTS` and reads `skills/roadmap/phases/refine.md` to execute Steps R1–R7 (mirroring `design/`'s thin-orchestrator → `phases/<phase>.md` pattern). Auto-detect via `_lib/phase-detect.sh` does **not** apply — refine requires the explicit flag, same as design's refine.
- **Refine lens fanout is three named auditors in one tool-call message.** `subagent_type` values MUST carry the `task:` plugin prefix: `task:audit-roadmap-coverage-auditor` / `-decomposition-auditor` / `-clarity-auditor`. Unprefixed names silently route to catch-all `claude`. The fanout shape mirrors `/task:build` audit phase Step 2b — same per-call prompt template, same `severity / category / location / problem / fix` output schema, same English field-key parser contract.
- **Refine writes to a sidecar log, never to a workspace.** Findings land in `.task/roadmap/<slug>.refine.md` as `## Iteration N` append-only blocks (same template as `audit.md`). Refine is roadmap-stage; there is no `.task/workspace/<task-id>/` and no `.task-current` involved.
- **Refine auto-fix is bounded ≤ 2 iterations and applies only `severity: high` findings.** Mirrors `/task:build` audit auto-fix loop. Medium/low findings stay `pending fix` in the sidecar log for the user to apply manually. Terminal status strings stay English regardless of `config.md` → "Language" (parser-stable): `--refine stopped: no high-severity remaining` or `--refine stopped: iteration limit (high-severity still pending — see <slug>.refine.md)`.
- **Refine never flips `- [ ]` → `- [x]`** — checkbox state is owned exclusively by `/task:ship` `close.sh:Step 1.5`. Refine rewrites item bodies, can renumber items + update `Dependencies:` lines when splitting compounds, but never touches checkbox state.

## `/task:auto-roadmap`

Shared mechanics (three Step 0 gates, `--items` grammar, lock shapes, failure protocol, cross-worktree safety) live in [auto-roadmap.md](auto-roadmap.md). Only orchestrator-specific invariants below.

- **Architectural premise — interactive runtime.** All LLM calls happen in the user's interactive Claude Code session (the driver's main thread, the item-runners it spawns, or their sub-subagents). Consequences: no background execution (session window must stay open), no `claude -p` subprocess. Model selection is **per-stage**, not session-wide: the item-runner (and the design-runner it spawns) inherit the parent-session model (user's `/model` choice); implement runs under each item's `plan.md → Implement-Model:` (`opus|sonnet|haiku`), passed by the item-runner as `Agent.model` override when spawning `auto-roadmap-build-runner`; the three audit lens auditors (`audit-{reuse,simplicity,clarity}-auditor`) pin `model: sonnet` in their own frontmatter and do **not** inherit. Full split table in [auto-roadmap.md § Per-stage model split](auto-roadmap.md).
- **Per-item architecture: one `auto-roadmap-item-runner` per item runs the whole cycle.** The driver spawns it and routes on its returned digest; the item-runner spawns everything else.
  - **`auto-roadmap-design-runner`** (spawned by item-runner) — design's open + blueprint phases, returns `OK: item #<N> "..." — plan.md ready, awaiting implement` or `FAIL at <stage>: ...`.
  - **The item-runner reads `plan.md → Implement-Model:`** (its Step 3) between the two child spawns. Missing/malformed value → fail-stop with an orchestrator-fail postmortem (validate.sh should have already caught it; surface as a bug rather than default silently).
  - **`auto-roadmap-build-runner`** (spawned by item-runner with the extracted `Agent.model`) — build's implement phase, returns `OK: item #<N> "..." — diff uncommitted, ready for audit` or `FAIL at implement: ...`. Only the post-open FAIL shape applies — the workspace subfolder always exists when build-runner is spawned.
  - **The item-runner runs `/task:build` audit phase + `/task:ship` itself** after build-runner OK. It fans out the three lens auditors directly — nested subagent spawning is supported, so the parallel three-lens view lives inside the item-runner rather than being forced onto an inline single-lens path.
  - **The item-runner returns a compact report-card digest** (last line parser-stable: `OK: item #<N> shipped (--next|--full) — <sha>` or `FAIL at <stage>: ...`). The digest carries `task_id:` (always) and `roadmap_mtime:` (only on `--next`); the driver greps both. `<stage>` enum: `design | lock | model-extract | implement | audit | ship`.
- **"Inline" means "read SKILL.md (and the relevant `phases/<phase>.md` companion) and execute its Steps", not "dispatch via the `Skill` tool".** Every pipeline skill carries `disable-model-invocation: true` by the universal invariant — the model (including a subagent) cannot dispatch them. The `auto-roadmap-item-runner` therefore reads the relevant SKILL.md/phase file and follows each Step directly for its audit + ship stages. Anyone editing the item-runner must preserve this — phrasing it as "invoke `/task:build` via Skill tool" reintroduces a contradiction with the universal `disable-model-invocation` invariant.
- **Single per-umbrella sentinel.** `workspace/<task-id>/auto.lock` is written by the **first item-runner** (its Step 2, gated on `is_first`) after its design-runner lands `.task-current` and the workspace subfolder. The on-disk file is the first persistent record + the cross-worktree mutex (Step 0 gate 3 scans `workspace/*/auto.lock`). Concrete shape in [artifact-contract.md](artifact-contract.md); invariants in [auto-roadmap.md § Sentinel shape](auto-roadmap.md#sentinel-shape--anchor-only).
- **Last worked item ships with `--full` (item-runner Step 6, Branch B).** The driver computes `is_last` by look-ahead over checkbox state (last run-set item still `- [ ]`), not "last entry in the run set" — a trailing already-`[x]` item is skipped, so the true last worked item must close the umbrella. No separate `chore-finalize` commit. Slug is auto-derived from `summary.md` so the last commit describes the last item, not a sweep chore. The `--full` close auto-marks the roadmap (Description non-empty gates auto-mark in `close.sh:Step 1.5` regardless of `FULL`), archives `task.md` alongside `plan/audit/summary.md`, removes the entire `workspace/<task-id>/` subfolder (sweeping the per-umbrella `auto.lock` with it), and removes `.task-current`. The slug `chore-finalize` is now reserved for **manual** recovery of an aborted run via `/task:ship --full chore-finalize` — the orchestrator never emits it.
- **Item count is effectively unbounded.** The per-item diff bundle + 3 lens-agent results live in the disposable item-runner context and are discarded on return; the driver accumulates only one compact digest per item. This greatly relaxes the old ~15 (Sonnet 200k) / ~25 (Opus 1M) auto-compact ceiling — a long run is now bounded by wall-clock, not the driver's context. Slice with `--items <range>` if you still want to bound one.
- **Status visibility is the item-runner's returned digest (relayed by the driver) + on-disk artifacts + `git`; `auto-error.log` on failure.** Live per-lens output streams in the item-runner's own subagent view, not the driver's transcript — the digest is the per-item record the driver prints.
- **Forward-only, strictly not for resume.** The `--from #<N>` flag is for resuming **a fresh run** that picks up where a previously cleaned-up failure left off; it is not an in-place resume.

## Shared prompt preamble

Three preamble profiles shared by all `skills/<name>/SKILL.md` files (and inherited by phase companion files at `skills/<name>/phases/<phase>.md`). **Prompt-layer only** — the runtime fail-closed gates live in `*-context.sh`, `validate.sh`, and `close.sh` and are NOT replaced by these profiles. Modifying these does not relax bash gates.

### Tier A — No code navigation

For: `/task:ship`, `validate`, `/task:auto-roadmap`, `/task:bootstrap`. Code work is delegated (subagents or none); the skill itself only touches `.task/` and git.

**Precondition (hard-stop).** `.task/config/config.md` must exist. If not — stop and tell the user: "Run `/task:bootstrap` first to initialize AI tools for the project." This is the **prompt-layer** reminder. The matching bash gate is in `commit-context.sh` / `auto-roadmap-context.sh` (each refuses non-zero exit on missing config) or in `close.sh` (same check at script entry). Skills without a context script run the check inline before any read. **Do not DRY the bash gate** — duplicated preconditions at the bash layer are the runtime guarantee.

**Constraint — no code navigation.** Allowed reads:

- `CLAUDE.md`, `.task/config/config.md`, `.task/workspace/<task-id>/*.md`, `.task-current`;
- `.task/log/**/*.md`, `.task/roadmap/*.md` (read-only for design's `--from`; in-place item-flip for `close.sh` auto-mark);
- `git` commands;
- the skill's own shell scripts under `${CLAUDE_SKILL_DIR}` (e.g. `close.sh`, `commit-context.sh`).

`/task:ship` additionally consumes the project commit-format doc (typically `CONTRIBUTING.md`) when `config.md` → "Commit Format" carries `**Source:** <path>` — `commit-context.sh` bundles it into the context block; the skill itself never opens the file.

**Forbidden.** MCP code-navigation tools, built-in `Grep`/`Glob`/`Read` against project source, opening any file outside the list above.

**Language.** See `.task/config/config.md` → "Language". Artifacts and user-facing dialog follow that value. **Parser-stable strings stay English regardless of language**: `workspace/<task-id>/auto.lock` field keys + values, `auto-error.log` contents, runner return strings (`OK: ... — diff uncommitted, ready for audit`, `FAIL at <stage>: ...`), `## Iteration {N}` headers, `Roadmap:` / `Source item:` header lines, the slug. `/task:ship`'s commit step additionally honors the commit-language directive inside `config.md` → "Commit Format".

### Tier B — MCP-first tooling

For: design's blueprint + refine phases, `/task:build` (both phases: implement, audit). These phases navigate and edit project code.

**Precondition (hard-stop).** Same as Tier A: `.task/config/config.md` must exist; otherwise stop with the `/task:bootstrap` redirect. Bash gate lives in the phase's context script when one exists (`audit-context.sh`) or in `validate.sh` invoked inline at Step 0 (orchestrator gates). The `PreToolUse` hook also runs `validate.sh all` before any Skill invocation in this tier.

**Constraint (critical) — MCP-first tooling.** Use EXCLUSIVELY MCP tools from `config.md` → "Code Navigation" / "Code Editing" in the priority order listed there. Built-in tools (`Edit`, `Read`, `Grep`, `Glob`, `ls`) are allowed ONLY when `config.md` explicitly lists them as fallback for the operation. If you choose a built-in tool over an available MCP tool — justify the reason in your reasoning before calling it.

**Forbidden.** Reading entire files when a symbol-scoped MCP call exists; running MCP code-navigation calls that are not in `config.md`'s priority list; bypassing fallback ordering.

Per-phase scope overrides on top of this tier:

- **design refine** — Read policy: reading **relevant symbol bodies** is permitted and expected (alternatives evaluation requires checking existing patterns). Do not read entire files.
- **build implement** — restrict edits to the symbols named in each `### Step N` → `Touches`; do not touch unrelated symbols in the same file unless `Goal` explicitly requires it.
- **build audit** — lens-agent calls run **read-only** (frontmatter-enforced); fixes happen in the orchestrator's main thread only (build SKILL.md Step 4), scope-gated by `_lib/touches-gate.sh`.

**Language.** See `.task/config/config.md` → "Language". `plan.md`, `audit.md`, `summary.md`, and all `## Decisions` follow the language of `task.md` Description. **Parser-stable strings stay English regardless**: `## Iteration {N}` headers, agent output field keys + the `high`/`med`/`low` enum, file paths, line numbers, `no findings` sentinels. Build's audit phase propagates the configured language into subagents via the explicit `--- Language ---` block in the per-call template.

### Tier C — Shallow scan

For: `/task:bootstrap`, `/task:roadmap`, design's idea phase, design's open phase **quick-draft path only**. These phases brainstorm, paraphrase, or set up configuration; they ground discussion in the project's stack without diving into source. Open's header-only path (`--idea` opt-out / empty fresh call) stays Tier A — no scan at all.

**Precondition (hard-stop).** `/task:bootstrap` is the only skill that **creates** `config.md` — it has no `config.md`-existence precondition. `/task:roadmap` and design's idea + open phases require `.task/config/config.md`; stop with the `/task:bootstrap` redirect otherwise. Idea phase carries an additional hard-stop on `.task/workspace/<task-id>/task.md` presence + valid header; `/task:roadmap` additionally refuses to overwrite an existing `.task/roadmap/<slug>.md` without confirmation.

**Constraint — shallow code navigation.** Allowed reads:

- `CLAUDE.md`, `.task/config/config.md`, `.task/workspace/<task-id>/*.md`, `.task/roadmap/*.md`;
- top-level directory listing (`ls`, `Glob` at depth 1–2);
- build/manifest files (`package.json`, `Cargo.toml`, `go.mod`, `pom.xml`, `build.gradle`, `pyproject.toml`, etc.);
- README at the repo root;
- `git log` (last 10–20 commits for context / commit-style detection);
- `docs/` top level + obvious entry points; `/task:roadmap` is the primary consumer here.

**Forbidden.** Reading source files (`*.ts`, `*.go`, `*.java`, `*.py`, `*.rs`, `*.kt`, `*.swift`, `*.cs`, `*.cpp`, `*.c`, `*.rb`, `*.php`, etc.) — even via `Read`. `Grep` over source. MCP code-navigation tools. Opening files inside source/test directories.

**Rationale.** The goal at this tier is a coarse mental model — "this is a Go backend with these top-level modules", "this initiative breaks into these phases that touch these documented systems" — not "this function does X". Detailed exploration is build/implement's job per atomic task.

**Language.** See `.task/config/config.md` → "Language". User-facing dialog (and the body of any Description written in this tier — idea architect mode, open quick-draft) follows the language of `task.md` Description or, when Description is empty / being bootstrapped, of `$ARGUMENTS`. **`/task:roadmap` is bilingual by structural contract** — section headers, table column names, per-task field labels, and the `### Context` / `### Goal` / `### Outcomes` / `### Acceptance criteria` / `### Spec references` sub-headings inside the `**Ready description:**` blockquote stay English regardless (parser contract with design's `--from` and `validate.sh`). Open's quick-draft sub-headers (`### Problem` / `### Outcome` / `### Scope` / `### Constraints`) follow the same English-skeleton convention as idea architect mode Step A.4 — they are content section markers, not parser contracts, but staying English keeps grep-style review consistent.
